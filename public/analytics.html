Architecture
How the analytics pipeline works

This platform uses Plausible Analytics — an open-source, privacy-respecting alternative to Google Analytics. The script is ~1KB, loads async, and doesn’t set any cookies.

Custom events are fired via plausible('Event Name', {props: {...}}) at meaningful interaction points. No third-party SDKs, no data brokers.

The job board uses GitHub Actions workflows — a weekly company discovery cron and a daily ATS status check. All job metadata flows into Supabase, which also stores user progress. No user data is shared with analytics.

Design Principles
How a PM thinks about analytics
🔒
Privacy by default
No cookies, no fingerprinting, no cross-site tracking. Data is aggregated at collection — there’s nothing to leak because nothing identifiable is stored.
🎯
Instrument what matters
Three custom events cover the full user journey: engagement (Day Completed), conversion (Checkout Started), and reach (Contact Click). Everything else is noise.
⚡
Performance first
The Plausible script is ~1KB and loads asynchronously. It doesn’t block rendering, doesn’t add GDPR banners, and doesn’t slow down the course experience.
📊
Separate product data from analytics
User progress, job applications, and notes live in Supabase — not in analytics. Analytics answers “what happened”; the product database answers “who did it.”
